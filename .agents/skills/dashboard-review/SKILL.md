# 大屏审查技能

## 触发条件

- 大屏开发完成，需要整体审查。
- 需要检查视觉还原度。
- 需要性能优化。
- 需要检查 3D 地图、图表、WebSocket 数据链路或 Figma 还原是否符合项目约束。
- 需要在交付前发现问题并直接修复。

## 审查原则

- 先列出明确问题，再给出修复建议。
- 能直接修复的问题应直接修复，不停留在建议层面。
- 不修改 `figma/` 目录下的导出物，除非用户明确要求重新提取或更新素材。
- 不得用 CSS、SVG、Canvas 或 AI 生成替代 Figma 已提取素材。
- 审查结论必须基于实际代码、实际截图、实际运行效果或明确文件内容，不得凭空推断。

## 审查清单

### 性能审查

- [ ] Three.js 场景是否有内存泄漏，`geometry`、`material`、`texture` 是否正确释放。
- [ ] Three.js 动画循环是否在组件卸载时停止。
- [ ] OrbitControls、事件监听器、Raycaster 相关监听是否正确清理。
- [ ] ECharts 实例是否在组件卸载时销毁。
- [ ] ECharts 是否避免重复 `init`，数据更新是否复用 `setOption`。
- [ ] WebSocket 连接是否按业务需要正确关闭或复用。
- [ ] WebSocket 重连、心跳和消息订阅是否会造成重复注册。
- [ ] 是否有不必要的 Vue 重渲染。
- [ ] 动画是否使用 `requestAnimationFrame` 或受控动画库。
- [ ] 资源是否按需加载，贴图、模型、GeoJSON 是否避免重复请求。

### 视觉还原审查

- [ ] 对照 Figma 设计截图检查面板布局。
- [ ] 检查颜色、字体、间距、层级是否与设计一致。
- [ ] 检查面板背景框是否使用真实提取素材，即 `figma/exports/`，而非 CSS 重新绘制。
- [ ] 检查 Figma 素材是否被拉伸、压缩、裁切错误或重复导入。
- [ ] 3D 地图风格是否与参考截图一致，包括视角、光照、材质、边界和辉光。
- [ ] 图表样式是否与大屏整体风格协调。
- [ ] 1920 x 1080 基准尺寸下是否布局准确。
- [ ] 常见桌面比例下是否无重叠、无溢出、无错位。

### 交互审查

- [ ] 3D 地图旋转、缩放、平移是否流畅。
- [ ] 地图 hover 高亮是否正常。
- [ ] 双击下钻是否正常工作。
- [ ] 面包屑回退是否正常。
- [ ] 资源点 hover / click 交互是否正常。
- [ ] 飞线、呼吸灯、粒子等动画是否稳定且不会累积实例。
- [ ] 图表数据更新是否实时。
- [ ] 图表点击、hover 或筛选是否能按需求联动地图。
- [ ] WebSocket 断开、重连、异常消息场景是否有可控表现。

### 代码质量审查

- [ ] 是否使用 `<script setup>` 语法。
- [ ] WebSocket 地址是否从环境变量读取。
- [ ] 是否遵循目录约定。
- [ ] 3D 逻辑是否封装在 `src/composables/` 中。
- [ ] ECharts 图表配置与业务数据是否分离。
- [ ] Pinia store 是否只存放跨组件共享状态。
- [ ] 是否存在硬编码接口地址、临时 mock、调试日志或未使用代码。
- [ ] 如果项目已启用 TypeScript，是否有必要的类型定义；如果未启用 TypeScript，不得为单个功能擅自引入 TS 构建改造。

## 输出

- 审查报告：列出问题、影响范围、严重程度和修复建议。
- 直接修复发现的问题，并说明改动文件。
- 对无法修复的问题说明阻塞原因和需要用户补充的信息。
- 将有效的开发模式沉淀到 `.agents/skills/` 中。
